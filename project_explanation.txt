1. WHAT IS THIS PROJECT?
================================================================================

Market Intelligence Scout is an AI-powered competitive intelligence system.

Given a company name (e.g. "OpenAI"), it:

  1.  Validates input via security guardrails (OWASP-compliant)
  2.  Generates smart search queries using an LLM
  3.  Searches the web via the Tavily API
  4.  Scrapes top articles using 3 strategies (BeautifulSoup, Newspaper3k, Playwright)
  5.  Filters out anything older than 7 days
  6.  Removes non-technical content
  7.  Scores each source's authority and credibility
  8.  Extracts structured technical features via LLM
  9.  Cross-verifies features across sources using SBERT embeddings
  10. Calculates confidence scores for each feature
  11. Generates an executive intelligence report with citations

The output is a structured briefing with verified technical features,
confidence scores, source citations, and an executive summary.

4. THE PIPELINE — 11-STAGE INTELLIGENCE WORKFLOW
================================================================================

The pipeline is a LangGraph StateGraph. Each node reads from and writes to
a shared GraphState dictionary. The pipeline runs asynchronously via Celery.

FLOW:  User Request → FastAPI → Celery Task → LangGraph Pipeline → Database → Response

────────────────────────────────────────────────────────────────────

STAGE 1: GUARDRAILS                          [nodes/guardrails.py]
────────────────────────────────────────────────────────────────────
Purpose:  Security firewall for input validation (OWASP-compliant)

  • Strips HTML tags, unescapes HTML entities, collapses whitespace
  • Validates name length (2–200 chars) and format (regex)
  • Blocks dangerous keywords (hack, exploit, jailbreak, etc.)
  • Enforces Redis-backed rate limiting (10 req / 60s)
  • Optional LLM semantic validation at temperature=0

  Writes → company_name (sanitised), error (if blocked)
  Routes → "search_agent" or "error_exit"

────────────────────────────────────────────────────────────────────
STAGE 2: SEARCH AGENT                    [agents/search_agent/]
────────────────────────────────────────────────────────────────────
Purpose:  Intelligent query planning + web discovery

  • LLM generates high-quality search queries
  • Tavily API executes web search
  • Filters: removes duplicates, skips low-value domains
  • Iterative loop (max 2 iterations) with critic feedback
  • Stores query memory in Redis

  Components: agent.py (orchestrator), planner.py (LLM queries),
              executor.py (Tavily), critic.py (quality check), memory.py

  Writes → search_results (URLs + snippets)

────────────────────────────────────────────────────────────────────
STAGE 3: SCRAPER AGENT                   [agents/scraper_agent/]
────────────────────────────────────────────────────────────────────
Purpose:  Extract clean article content from URLs

  For each URL (processed in parallel via ThreadPoolExecutor):
    1. Cache check → skip if already scraped
    2. LLM selects strategy → beautifulsoup / newspaper3k / playwright
    3. Execute with fallback chain (max 3 failures per URL)
    4. Clean content → extract title, text, publish date
    5. LLM critic validates technical relevance
    6. Cache result in Redis

  Components: agent.py, planner.py, critic.py, memory.py
              tools/ → bs4.py, newspaper.py, playwright.py

  Writes → scraped_articles

────────────────────────────────────────────────────────────────────
STAGE 4: DATE VALIDATION                [nodes/date_validation.py]
────────────────────────────────────────────────────────────────────
Purpose:  Filter out stale content (>7 days old)

  • Parses ISO-8601 dates, compares against recency window
  • No date → conservatively discarded
  • Audit trail stored in Redis

  Writes → filtered_results, discarded_urls
  Routes → "content_filter" or "all_expired"

────────────────────────────────────────────────────────────────────
STAGE 5: CONTENT FILTER                  [nodes/content_filter.py]
────────────────────────────────────────────────────────────────────
Purpose:  Remove non-technical / irrelevant articles

  • Keyword density analysis (API, SDK, model, benchmark, etc.)
  • Rejects marketing, job postings, investor relations content

  Writes → filtered_results (overwritten)
  Routes → "authority_check" or "no_technical"

────────────────────────────────────────────────────────────────────
STAGE 6: AUTHORITY CHECK                [nodes/authority_check.py]
────────────────────────────────────────────────────────────────────
Purpose:  Score and rank sources by credibility

  • Domain reputation tiers (official, tech publication, .edu/.gov)
  • Enriches each article with authority_score (0.0–1.0)
  • Sorts results by authority (highest first)

  Writes → filtered_results (enriched with scores)

────────────────────────────────────────────────────────────────────
STAGE 7: FEATURE EXTRACTION          [nodes/feature_extraction.py]
────────────────────────────────────────────────────────────────────
Purpose:  Extract structured technical features via LLM

  Per article, extracts:
    • feature_summary — detailed 2–3 sentence description
    • category — model_release | api_update | performance | capability
    • metrics — quantitative measurements
    • evidence — direct quotes from the article

  Writes → extracted_features
  Routes → "verification" or "no_features"

────────────────────────────────────────────────────────────────────
STAGE 8: VERIFICATION                    [nodes/verification.py]
────────────────────────────────────────────────────────────────────
Purpose:  Cross-reference features using SBERT semantic similarity

  • Loads Sentence-BERT model (all-MiniLM-L6-v2)
  • Encodes features into 384-dim embeddings
  • Clusters features with cosine similarity ≥ 0.85
  • Merges clusters into verified features with source_count

  Writes → verified_features

────────────────────────────────────────────────────────────────────
STAGE 9: CONFIDENCE SCORING                  [nodes/scoring.py]
────────────────────────────────────────────────────────────────────
Purpose:  Calculate final trust score for each feature

  Three scoring signals (each 0.0 → 1.0):
    a) Recency   — age-based decay (today=1.0, 7d=0.5, >14d=0.3)
    b) Verification — source count (logarithmic, 5+=~1.0)
    c) Authority — direct pass-through of source score

  Final: confidence = 0.3×recency + 0.4×verification + 0.3×authority

  Writes → scored_features

────────────────────────────────────────────────────────────────────
STAGE 10: SYNTHESIS                         [nodes/synthesis.py]
────────────────────────────────────────────────────────────────────
Purpose:  Generate the final executive intelligence report via LLM

  Produces:
    • executive_summary — 3–5 paragraph strategic overview
    • features — ranked list with titles, descriptions, confidence
    • all_sources — complete URL list with citations

  Includes deterministic fallback if LLM fails.

  Writes → synthesis_report

────────────────────────────────────────────────────────────────────
ERROR / EARLY-EXIT NODES
────────────────────────────────────────────────────────────────────
5 graceful degradation paths:

  • no_results   → "No search results found for '{company}'"
  • no_articles  → "All URLs failed to scrape"
  • all_expired  → "All articles are older than 7 days"
  • no_technical → "No technical content found"
  • no_features  → "No extractable features found"

  All → error_exit → END (returns consistent response shape)

5. ASYNC ARCHITECTURE — CELERY INTEGRATION
================================================================================

The pipeline runs asynchronously to avoid blocking the API:

  1. Frontend sends POST /run-agent {"company_name": "OpenAI"}
  2. FastAPI enqueues a Celery task → returns task_id immediately
  3. Celery worker picks up the task and runs the LangGraph pipeline
  4. Frontend polls GET /task/{task_id} every 2 seconds
  5. When complete, the result is returned with the full report

CELERY CONFIGURATION:
  • Broker:  Redis db=1  (redis://redis:6379/1)
  • Results: Redis db=2  (redis://redis:6379/2)
  • Cache:   Redis db=0  (default)
  • Concurrency: 2 workers, max 50 tasks per child process
  • Monitoring: Flower dashboard at http://localhost:5555

CONCURRENCY CONTROL:
  • Global threading.Semaphore(3) limits concurrent Playwright instances
  • Prevents memory exhaustion under multi-user load

7. THE DATABASE — POSTGRESQL + SQLALCHEMY
================================================================================

4 tables defined across database/models.py and scheduled_job_model.py:

  TABLE: competitors
    ├── id            (PK, auto-increment)
    ├── name          (unique, indexed)
    ├── industry      (default: "technology")
    └── created_at    (UTC timestamp)

  TABLE: reports
    ├── id                (PK, auto-increment)
    ├── competitor_id     (FK → competitors.id)
    ├── executive_summary (Text)
    ├── total_sources     (Integer)
    ├── total_features    (Integer)
    ├── all_sources       (JSON array of URLs)
    ├── metadata          (JSON)
    └── created_at        (UTC timestamp)

  TABLE: features
    ├── id                (PK, auto-increment)
    ├── competitor_id     (FK → competitors.id)
    ├── report_id         (FK → reports.id)
    ├── feature_title     (Short title)
    ├── feature_text      (2–3 sentence summary)
    ├── description       (Full description)
    ├── category          (model_release | api_update | performance | capability)
    ├── confidence_score  (Float 0.0–1.0)
    ├── source_count      (Integer)
    ├── source_url        (String)
    ├── evidence          (Text — direct quotes)
    ├── metrics           (JSON array)
    └── created_at        (UTC timestamp)

  TABLE: scheduled_jobs
    ├── id            (PK, auto-increment)
    ├── report_id     (FK → reports.id)
    ├── scheduled_at  (DateTime)
    ├── status        (pending | running | done | failed)
    ├── email         (notification email)
    └── created_at    (UTC timestamp)

  Relationships:
    Competitor → has many → Reports → has many → Features
    Report → has many → ScheduledJobs

  Deletion order (FK-safe):
    ScheduledJob → Feature → Report → Competitor

9. THE LLM CLIENT — NVIDIA NIM
================================================================================

Defined in llm/nvidia_client.py:

  • Uses OpenAI Python SDK pointed at NVIDIA's API
  • Base URL: https://integrate.api.nvidia.com/v1
  • Model: meta/llama-3.3-70b-instruct
  • Parameters: temperature=0.2, top_p=0.7, max_tokens=4096
  • Singleton pattern: one client reused across all requests
  • Retry logic: exponential backoff (up to 3 retries)
  • Thread-safe: Semaphore-gated to prevent rate limit exhaustion
  • Prometheus metrics: call count, latency, token usage per agent

14. SECURITY IMPLEMENTATION (OWASP)
================================================================================

  A03 — INJECTION PREVENTION:
    • HTML tag stripping + entity unescaping in guardrails
    • Regex format validation on all inputs
    • Blocked keyword detection (hack, exploit, jailbreak, etc.)
    • Pydantic schema validation on all API payloads

  A05 — SECURITY MISCONFIGURATION:
    • All secrets in .env file, never hardcoded
    • Pydantic BaseSettings reads from environment

  A07 — XSS PREVENTION:
    • Input sanitisation + structured JSON responses
    • No raw HTML in API output

  A10 — SSRF PREVENTION:
    • Domain allowlist in config.py
    • Prefix-based allowlist (docs.*, developer.*, blog.*)
    • Every scraped URL validated against the allowlist

  RATE LIMITING:
    • Redis-backed counter: 10 requests per 60-second window
    • Enforced in guardrails node before any LLM call

15. DATA FLOW — COMPLETE REQUEST LIFECYCLE
================================================================================

  User types "OpenAI" → clicks "Analyze"
    ↓
  Frontend sends POST /run-agent {"company_name": "OpenAI"}
    ↓
  FastAPI enqueues Celery task → returns {"task_id": "abc-123"}
    ↓
  Frontend polls GET /task/abc-123 every 2 seconds
    ↓
  Celery worker picks up the task
    ↓
  LangGraph pipeline invoked: graph.invoke({"company_name": "OpenAI"})
    ↓
  Stage 1:  Guardrails → sanitise, validate, rate-limit check
  Stage 2:  Search Agent → LLM generates queries → Tavily returns URLs
  Stage 3:  Scraper Agent → parallel scraping → 10 articles extracted
  Stage 4:  Date Validation → filter to ≤7 days → 7 articles pass
  Stage 5:  Content Filter → keep technical only → 5 articles remain
  Stage 6:  Authority Check → rank by credibility
  Stage 7:  Feature Extraction → LLM extracts 12 raw features
  Stage 8:  Verification → SBERT clusters → 8 unique features
  Stage 9:  Scoring → confidence = 0.3×recency + 0.4×verify + 0.3×authority
  Stage 10: Synthesis → LLM generates executive summary + report
    ↓
  Pipeline complete → result stored in Celery backend
    ↓
  FastAPI saves report + features to PostgreSQL
    ↓
  Frontend receives result via polling → renders report
