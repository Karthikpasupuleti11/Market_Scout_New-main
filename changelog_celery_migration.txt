MARKET SCOUT - CELERY MIGRATION CHANGELOG
=========================================

Date: May 17, 2026

OBJECTIVE ACHIEVED:
-------------------
Transitioned the Market Intelligence Scout backend from a synchronous, blocking architecture to a production-grade, asynchronous architecture using Celery and Redis. 
This prevents FastAPI worker lockups, prevents memory explosion from concurrent Playwright browser instances, and allows the system to scale horizontally under heavy load.

KEY ARCHITECTURAL CHANGES:
--------------------------
1. Celery Offloading: The entire LangGraph pipeline execution (which takes 60-180 seconds) is now pushed to background Celery workers.
2. Non-blocking API: The `/run-agent` endpoint now returns a `task_id` in milliseconds instead of holding the HTTP connection open.
3. Frontend Polling: The frontend now safely polls `/task/{id}` every 2 seconds for status updates instead of relying on fragile Server-Sent Events (SSE).
4. Concurrency Safety: Introduced a global threading Semaphore to limit Playwright Chromium instances to 3 concurrent browsers, preventing RAM exhaustion.
5. Observability: Integrated Flower for real-time monitoring of Celery tasks, queues, and worker health.
6. DB Isolation: Configured Celery to use Redis `db=1` for its broker and `db=2` for task results, ensuring no collision with the application's cache on `db=0`.

FILES ADDED:
------------
1. `celery_app/__init__.py`
   - Created the Celery application factory.
   - Configured Redis broker (db=1) and result backend (db=2).
   - Set up worker behavior: JSON serialization, worker recycling (max 50 tasks per child), and memory limits.

2. `celery_app/tasks.py`
   - Created `run_pipeline_task` to execute the LangGraph pipeline in the background.
   - Implemented state tracking to push real-time node progress to Celery's meta state.
   - Handled Prometheus metric recording and DB persistence safely within the worker context.

FILES MODIFIED:
---------------
1. `app/config.py`
   - Added `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`, and `PLAYWRIGHT_MAX_CONCURRENT` to the settings schema.

2. `agents/scraper_agent/tools/playwright.py`
   - Added a `threading.Semaphore(limit)` to cap concurrent Chromium browsers to 3 (or the value set in config).

3. `app/main.py`
   - Refactored `POST /run-agent` to enqueue the Celery task and return the `task_id` immediately.
   - Added `GET /task/{task_id}` to allow the frontend to poll for task progress and results.

4. `docker-compose.yaml`
   - Added `celery_worker` service with prefork concurrency=2.
   - Added `flower` monitoring service on port 5555.
   - Injected Celery environment variables into the main app and worker.

5. `Dockerfile`
   - Installed Playwright Chromium system dependencies (libnss3, libgbm1, libasound2, etc.) using `apt-get` so browsers can launch successfully inside Docker.
   - Added `playwright install chromium` command.

6. `requirements.txt`
   - Added `celery==5.4.0`
   - Added `flower==2.0.1`

7. `.env`
   - Added `CELERY_BROKER_URL` and `CELERY_RESULT_BACKEND` default values.
   - Added `PLAYWRIGHT_MAX_CONCURRENT=3`.

8. `monitoring/prometheus.yml`
   - Added the Flower metrics endpoint as a scrape target to capture Celery worker metrics alongside FastAPI.

9. `frontend/src/api.js`
   - Renamed `runPipeline` to `submitPipeline` (removes signal abort logic since it returns instantly).
   - Added `getTaskStatus(taskId)` function to poll the backend.

10. `frontend/src/contexts/PipelineContext.jsx`
    - Replaced the `runPipelineSSE` streaming logic with a robust `while` loop that polls `getTaskStatus` every 2 seconds.
    - Preserved all real-time UI animation capabilities while drastically improving connection stability.
